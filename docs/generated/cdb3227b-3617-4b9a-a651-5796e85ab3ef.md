# drawio_kato システム設計書

## プロジェクト概要

**プロジェクト名**: drawio_kato  
**リポジトリ**: https://github.com/thippi555/drawio_kato  
**AWSリージョン**: ap-northeast-1  
**ランタイム**: Python 3.12  
**IaC**: Terraform  

## システムアーキテクチャ

### 概要

drawio_katoは、ユーザーのリクエストに基づいてAI（Claude Haiku）を活用し、設計ドキュメントとDrawio形式のアーキテクチャ図を自動生成するサーバレスシステムです。

### AWSサービス構成

#### 1. API Gateway
- **エンドポイント**: POST /tasks
- **役割**: ユーザーリクエストの受け入れ、Lambda関数への転送
- **認証**: 個人利用のため基本認証またはAPIキー

#### 2. Lambda関数
- **関数名**: drawio-kato-task-processor
- **ランタイム**: Python 3.12
- **役割**: 
  - リクエスト検証
  - タスク情報をDynamoDBに記録
  - Step Functionsの実行開始
  - レスポンス返却

#### 3. Step Functions (State Machine)
- **ステートマシン名**: drawio-kato-task-flow
- **役割**: 
  - タスク処理フロー制御
  - Bedrockへのプロンプト送信
  - 出力ファイルの生成と保存
  - エラーハンドリング

#### 4. Amazon Bedrock
- **モデルID**: global.anthropic.claude-haiku-4-5-20251001-v1:0
- **役割**: 
  - 設計ドキュメント（Markdown）の生成
  - Drawio XML形式のアーキテクチャ図定義の生成
  - 構造化JSON形式の成果物生成

#### 5. Amazon S3
- **バケット名**: drawio-kato-artifacts
- **ストレージプレフィックス**:
  - `tasks/`: タスク定義ファイル
  - `outputs/`: 生成された成果物
  - `prompts/`: Bedrockへのプロンプトテンプレート
  - `logs/`: 処理ログ
- **出力ファイル**:
  - `design.md`: 設計書（Markdown形式）
  - `architecture.drawio`: アーキテクチャ図（XML形式）
  - `artifact.json`: 構造化成果物（JSON形式）

#### 6. Amazon DynamoDB
- **テーブル名**: ai_agent_tasks
- **主キー**: task_id (String)
- **属性**:
  - task_id: タスク一意識別子
  - user_id: ユーザー識別子
  - request_content: ユーザーリクエスト内容
  - status: タスク状態（pending, processing, completed, failed）
  - created_at: 作成日時
  - updated_at: 更新日時
  - execution_arn: Step Functions実行ARN
  - output_s3_path: S3出力パス

## 処理フロー

### リクエスト処理フロー

1. **ユーザーリクエスト**
   - API Gateway (POST /tasks) にJSON形式でリクエスト送信
   - リクエスト例: `{"design_request": "マイクロサービスアーキテクチャの設計"}`

2. **Lambda処理（drawio-kato-task-processor）**
   - リクエスト検証
   - task_idを生成
   - DynamoDBにタスク記録（status: pending）
   - Step Functionsの実行を開始
   - task_idをレスポンスで返却

3. **Step Functions実行（drawio-kato-task-flow）**
   - タスク情報をDynamoDBから取得
   - ステータスを「processing」に更新
   - Bedrockへプロンプト送信
   - 3つの並列タスク実行:
     - 設計書Markdown生成
     - Drawio XML生成
     - 構造化JSON生成
   - 生成ファイルをS3に保存
   - DynamoDBのステータスを「completed」に更新
   - エラー発生時はステータスを「failed」に更新

4. **Bedrock処理**
   - Claude Haikuモデルを使用
   - プロンプトテンプレートをS3から取得
   - ユーザーリクエストを組み込んでプロンプト生成
   - 3つの異なるプロンプトで並列実行
   - 生成結果を返却

5. **S3保存**
   - outputs/{task_id}/design.md
   - outputs/{task_id}/architecture.drawio
   - outputs/{task_id}/artifact.json
   - logs/{task_id}/execution.log

## セキュリティ考慮事項

- IAM ロールによる最小権限の原則
- Lambda関数のタイムアウト設定（300秒推奨）
- S3バケットの暗号化有効化
- DynamoDBのポイントインタイム復旧有効化
- API Gatewayのレート制限設定

## コスト最適化

- Lambda: 従量課金（個人利用のため低コスト）
- DynamoDB: オンデマンド課金モード
- S3: 標準ストレージクラス
- Bedrock: トークン単位の従量課金
- Step Functions: 状態遷移単位の課金
